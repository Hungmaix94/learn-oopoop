## 01. CSDL Là Gì? (Mô Hình Hóa Thực Tế)

### Bản Chất Số Hóa

CSDL không chỉ là kho lưu trữ, nó là một **bản sao kỹ thuật số của thế giới thực**.

Chúng ta lược bỏ các chi tiết thừa (bụi bẩn, màu sơn hỏng) và giữ lại các **Thuộc tính cốt lõi** để máy tính có thể quản lý.

### Ví Dụ: Mô Hình Xe Máy

| Ngoài đời | Trong CSDL |
|:---|:---|
| Một chiếc xe máy thực tế có vô vàn chi tiết | Gói gọn trong thực thể `[XE_MAY]` với các thuộc tính: BienSo, NhanHieu, MauSac, NamSX |

### Ví Dụ Mô Hình Hóa: Bệnh Viện

| 🏥 Physical Reality | 💾 Database Mapping |
|:---|:---|
| Anh Nguyễn Văn A đang khám bệnh | `Table BenhNhan`: ID, Ten, BenhLy |
| Bác sĩ Lê Thị B đang kê đơn | `Table BacSi`: ID, ChuyenKhoa |
| Phòng bệnh 102 đang trống | `Table LichHen`: PatientID, DoctorID, Time |

### Khi Không Có CSDL, Chúng Ta Dùng Gì?

Trước khi CSDL ra đời hoặc với các ứng dụng siêu nhỏ, chúng ta thường lưu dữ liệu vào các **File văn bản** (như `.txt`, `.csv`, `.json`, `.excel`).

**Vấn đề của việc dùng File:**
*   **Tốc độ:** Tìm một dòng trong file 1 triệu dòng rất chậm (phải đọc từ đầu đến cuối).
*   **Bảo mật:** Khó phân quyền ai được xem cột nào, dòng nào.
*   **Đồng thời:** Hai người cùng mở một file Excel và nhấn Save cùng lúc sẽ bị ghi đè dữ liệu.
*   **Ràng buộc:** Không thể ngăn chặn việc nhập "Tuổi = -5" vào một ô trong File một cách tự động và chặt chẽ.

### 5 Ưu Điểm Vượt Trội Của CSDL

1.  **Truy xuất cực nhanh:** Nhờ cơ chế **Index** (chỉ mục), tìm kiếm 1 trong 10 triệu bản ghi chỉ mất vài mili giây.
2.  **Quản lý quan hệ phức tạp:** Dễ dàng liên kết các đối tượng (Người dùng -> Đơn hàng -> Sản phẩm).
3.  **Toàn vẹn dữ liệu:** Đảm bảo dữ liệu luôn đúng quy định (Tính nhất quán - Consistency).
4.  **Hỗ trợ đa người dùng:** Hàng ngàn người có thể đọc/ghi dữ liệu cùng lúc mà không gây xung đột (Tính độc lập - Isolation).
5.  **Bảo mật & Phân quyền:** Cho phép cấu hình chi tiết: "Nhân viên kho chỉ được xem tồn kho, không được xem lương giám đốc".

> **➔ CSDL giúp quản lý hàng triệu bản ghi chỉ với vài mili giây truy vấn.**

## 02. Tính Chất ACID

> **4 Vệ Sĩ Bảo Vệ Toàn Vẹn Dữ Liệu**

### A - Atomicity (Tính Nguyên Tố)

**Quy tắc: "Tất cả hoặc không gì"**

Giao dịch phải thành công 100% hoặc thất bại 100%. Không bao giờ có trạng thái dở dang.

**Case Study: Chuyển Tiền**
```
1. Trừ tiền tài khoản A (Thành công)
2. Hệ thống sập nguồn đột ngột
3. Chưa cộng tiền cho tài khoản B
```
➔ **ACID HÀNH ĐỘNG**: Tự động hoàn tiền cho A (Rollback).

---

### C - Consistency (Tính Nhất Quán)

**Quy tắc: "Luôn đúng luật"**

Dữ liệu phải chuyển từ trạng thái hợp lệ này sang trạng thái hợp lệ khác theo đúng ràng buộc hệ thống.

**Ví Dụ: Ràng Buộc Tuổi**
```
Hệ thống quy định: [Tuổi > 0]
Nếu người dùng nhập: Tuổi = -5
```
➔ **CSDL HÀNH ĐỘNG**: Từ chối ghi dữ liệu để bảo vệ tính nhất quán.

---

### I - Isolation (Tính Độc Lập)

**Quy tắc: "Đừng chen lấn"**

Hai giao dịch diễn ra đồng thời không được phép nhìn thấy dữ liệu trung gian của nhau.

**Ví Dụ: Mua Vé Phim**
```
Người A và B cùng nhấn "Mua" chiếc ghế cuối cùng.
```
➔ **CSDL HÀNH ĐỘNG**: Khóa ghế cho người nhanh hơn 0.001s, người sau báo "Hết chỗ".

---

### D - Durability (Tính Bền Vững)

**Quy tắc: "Ghi bút bi"**

Một khi hệ thống đã báo "Giao dịch thành công", dữ liệu sẽ nằm vĩnh viễn trong ổ cứng dù có sập nguồn ngay sau đó.

**Ví Dụ: Thanh Toán**
```
App báo "Bạn đã thanh toán thành công"
Server cháy ngay lập tức
```
➔ **KẾT QUẢ**: Khi server hồi phục, đơn hàng của bạn vẫn tồn tại an toàn.

## 03. 4 Giai Đoạn Thiết Kế Chuẩn

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│  Phân Tích │ →  │ Quan Niệm  │ →  │   Logic    │ →  │  Vật Lý    │
├────────────┤    ├────────────┤    ├────────────┤    ├────────────┤
│ Thu thập   │    │ Vẽ sơ đồ   │    │ Xây dựng   │    │ Cài đặt &  │
│ yêu cầu    │    │ thực thể   │    │ cấu trúc   │    │ Tối ưu     │
│ nghiệp vụ  │    │ ERD        │    │ Bảng & Khóa│    │ Index/     │
│            │    │            │    │            │    │ Server     │
└────────────┘    └────────────┘    └────────────┘    └────────────┘
```

### GĐ 1: Phân Tích Yêu Cầu
- Thu thập yêu cầu từ khách hàng
- Xác định các nghiệp vụ cần số hóa
- Liệt kê các đối tượng và mối quan hệ

### GĐ 2: Thiết Kế Quan Niệm

Đây là giai đoạn vẽ **"bản thiết kế" (Blueprint)**. Sử dụng sơ đồ **ERD** để liên kết các đối tượng.

**Ví Dụ: Hệ thống Grab**
```
[TaiXe] ──1:N──► [ChuyenDi]
[KhachHang] ──1:N──► [ChuyenDi]
```
➔ Bản vẽ này giúp lập trình viên và khách hàng thống nhất logic.

### GĐ 3: Thiết Kế Logic (Cấu Trúc)

| Tên Bảng | Khóa Chính (PK) | Khóa Ngoại (FK) | Mô tả |
|:---|:---|:---|:---|
| `NguoiDung` | UserID | - | Lưu thông tin cá nhân |
| `DonHang` | OrderID | UserID | Liên kết đơn với người dùng |
| `ChiTietDon` | ID | OrderID, ProductID | Lưu số lượng từng món |

➔ Thiết kế **khóa ngoại (FK)** giúp dữ liệu không bị "mồ côi" (không có chủ).

### GĐ 4: Thiết Kế Vật Lý
- Cài đặt DBMS (MySQL, PostgreSQL, SQL Server)
- Tối ưu Index
- Cấu hình Server

## 04. Cú Pháp SQL Cơ Bản (CRUD)

### Create (Thêm mới)
```sql
INSERT INTO NguoiDung (ID, Ten, Email)
VALUES (1, 'Nguyen Van A', 'a@email.com');
```

### Read (Truy vấn)
```sql
SELECT * FROM NguoiDung WHERE ID = 1;

-- Truy vấn có điều kiện
SELECT Ten, Email FROM NguoiDung 
WHERE Tuoi > 18 
ORDER BY Ten ASC;
```

### Update (Cập nhật)
```sql
UPDATE NguoiDung 
SET Email = 'new@email.com' 
WHERE ID = 1;
```

### Delete (Xóa)
```sql
DELETE FROM NguoiDung WHERE ID = 1;
```

## 05. Index & Tối Ưu Hiệu Năng

### Chỉ Mục (Index) Là Gì?

Hãy tưởng tượng bạn tìm một từ trong cuốn từ điển 1000 trang:

| Không có Index | Có Index |
|:---|:---|
| Lật từng trang (Full Table Scan) | Tìm ngay đến trang chứa từ đó |
| ❌ **Cực chậm** | ✅ **Tức thì** |

### Cú Pháp Tạo Index

```sql
-- Tạo index cho cột Email
CREATE INDEX idx_email ON KhachHang(Email);
```

➔ Giúp tìm kiếm Email trong **1 triệu khách hàng** chỉ mất **0.001 giây**.

### Khi Nào Nên Dùng Index?

| ✅ Nên dùng Index | ❌ Không nên dùng |
|:---|:---|
| Cột hay dùng trong `WHERE` | Bảng có ít dữ liệu |
| Cột dùng trong `JOIN` | Cột hay bị UPDATE |
| Cột dùng trong `ORDER BY` | Cột có giá trị trùng lặp nhiều |

## 06. Bảo Mật: Chống SQL Injection

### Lỗ Hổng Chết Người

Hacker chèn mã SQL độc hại vào ô tìm kiếm để xóa sạch CSDL của bạn.

**Ví dụ tấn công:**
```sql
-- Input độc hại từ hacker
' OR 1=1; DROP TABLE NguoiDung; --

-- Query bị tiêm nhiễm
SELECT * FROM NguoiDung WHERE Ten = '' OR 1=1; DROP TABLE NguoiDung; --'
```

### Giải Pháp: Parameterized Query

Sử dụng các **tham số ẩn** thay vì nối chuỗi trực tiếp.

```typescript
// ❌ SAI - Dễ bị SQL Injection
const query = `SELECT * FROM Users WHERE name = '${userInput}'`;

// ✅ ĐÚNG - Parameterized Query
const query = 'SELECT * FROM Users WHERE name = ?';
db.execute(query, [userInput]);
```

## 07. Chuẩn Hóa (Normalization)

### 1NF - First Normal Form

**Quy tắc**: Mỗi ô chỉ chứa **1 giá trị duy nhất**. Không lưu danh sách trong 1 ô.

```
❌ Vi phạm 1NF:
┌─────┬────────────────────┐
│ ID  │ SoDienThoai        │
├─────┼────────────────────┤
│ 1   │ 0901, 0902, 0903   │  ← Nhiều giá trị trong 1 ô
└─────┴────────────────────┘

✅ Đạt 1NF:
┌─────┬─────────────┐
│ ID  │ SoDienThoai │
├─────┼─────────────┤
│ 1   │ 0901        │
│ 1   │ 0902        │
│ 1   │ 0903        │
└─────┴─────────────┘
```

### 2NF - Second Normal Form

**Quy tắc**: Phải có 1NF + Mọi cột dữ liệu phải **phụ thuộc hoàn toàn vào Khóa chính**.

```
❌ Vi phạm 2NF:
┌──────────┬──────────┬─────────────┬─────────────┐
│ OrderID  │ ProductID│ ProductName │ Quantity    │
├──────────┼──────────┼─────────────┼─────────────┤
│ 1        │ P001     │ iPhone 15   │ 2           │
└──────────┴──────────┴─────────────┴─────────────┘
              ↑
        ProductName phụ thuộc vào ProductID, không phải toàn bộ PK

✅ Đạt 2NF: Tách thành 2 bảng
- ChiTietDon(OrderID, ProductID, Quantity)
- SanPham(ProductID, ProductName)
```

### 3NF - Third Normal Form

**Quy tắc**: Loại bỏ **phụ thuộc bắc cầu**. Không tính toán cột này từ cột kia.

```
❌ Vi phạm 3NF:
┌──────────┬──────────┬───────────────┐
│ DonGia   │ SoLuong  │ ThanhTien     │
├──────────┼──────────┼───────────────┤
│ 100      │ 5        │ 500           │  ← Thừa! 500 = 100 × 5
└──────────┴──────────┴───────────────┘

✅ Đạt 3NF: Bỏ cột ThanhTien, tính toán khi cần
SELECT DonGia * SoLuong AS ThanhTien FROM ...
```

## 📚 Thuật Ngữ CSDL (Glossary)

| Tiếng Anh | Tiếng Việt | Giải thích |
|:---|:---|:---|
| Database Management System (DBMS) | Hệ quản trị CSDL | Phần mềm điều khiển (MySQL, SQL Server, PostgreSQL) |
| Primary Key (PK) | Khóa chính | Mã định danh duy nhất (ID) |
| Foreign Key (FK) | Khóa ngoại | Tham chiếu đến PK của bảng khác |
| Schema | Lược đồ | Bản đồ cấu trúc của toàn hệ thống |
| Constraint | Ràng buộc | Các luật lệ (VD: Giá không được < 0) |
| Index | Chỉ mục | Cấu trúc tăng tốc truy vấn |
| Transaction | Giao dịch | Một đơn vị công việc hoàn chỉnh |
| Query | Truy vấn | Câu lệnh yêu cầu dữ liệu |
| Normalization | Chuẩn hóa | Quy trình loại bỏ dư thừa dữ liệu |
| ERD | Sơ đồ thực thể | Bản vẽ mô tả quan hệ giữa các bảng |
