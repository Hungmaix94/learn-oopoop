# OOP vs FP: Cuộc Chiến Tư Duy Qua Các Ví Dụ E-commerce Thực Tế

> Một phân tích chuyên sâu so sánh **Lập trình hướng đối tượng (OOP)** và **Lập trình hàm (FP)** trong bối cảnh phát triển ứng dụng E-commerce bằng **TypeScript**.

---

## 1. Giới Thiệu

Trong phát triển phần mềm hiện đại, đặc biệt là với TypeScript, chúng ta thường thấy sự kết hợp của cả hai mô hình. Tuy nhiên, hiểu rõ sự khác biệt cốt lõi sẽ giúp bạn chọn công cụ đúng cho từng vấn đề.

| Đặc điểm | OOP (Object-Oriented Programming) | FP (Functional Programming) |
|:---|:---|:---|
| **Đơn vị chính** | Class & Object | Function |
| **Quản lý trạng thái** | Mutable (Trạng thái thay đổi bên trong object) | Immutable (Tạo bản sao mới, không đổi trạng thái cũ) |
| **Dữ liệu & Hành vi** | Gắn liền nhau (Encapsulation) | Tách biệt (Data structures + Pure functions) |
| **Code sharing** | Inheritance (Kế thừa) | Composition (Hợp thành) |

Dưới đây là 4 bài toán E-commerce điển hình được giải quyết bằng cả hai cách.

---

## 2. Bài Toán 1: Quản Lý Sản Phẩm (Data & Behavior)

**Yêu cầu:** Mô hình hóa một sản phẩm và chức năng cập nhật giá, đổi tên.

### 🏛️ Cách tiếp cận OOP: Encapsulation
Dữ liệu và hàm đi chung với nhau. Class chịu trách nhiệm bảo vệ dữ liệu của nó.

```typescript
// OOP: Class Product giữ data và method
class Product {
  constructor(
    private _id: string,
    private _name: string,
    private _price: number
  ) {}

  // Encapsulation: Logic validation nằm ngay trong class
  setPrice(newPrice: number) {
    if (newPrice < 0) throw new Error("Price cannot be negative");
    this._price = newPrice; // ⚠️ Mutation: Thay đổi trực tiếp
  }

  getName() { return this._name; }
  getPrice() { return this._price; }
}

const p1 = new Product("1", "Laptop", 1000);
p1.setPrice(1200); // p1 đã bị thay đổi (Mutation)
```

### 🧪 Cách tiếp cận FP: Separation
Dữ liệu chỉ là dữ liệu (Interface). Hàm là các "nhà máy" biến đổi dữ liệu đầu vào thành dữ liệu mới.

```typescript
// FP: Data riêng biệt
type Product = {
  readonly id: string;
  readonly name: string;
  readonly price: number;
};

// Pure Function: Nhận vào cũ -> Trả về NGUYÊN VẸN cái mới
const updatePrice = (product: Product, newPrice: number): Product => {
  if (newPrice < 0) throw new Error("Price cannot be negative");
  return { ...product, price: newPrice }; // ✅ Immutability
};

const p1: Product = { id: "1", name: "Laptop", price: 1000 };
const p2 = updatePrice(p1, 1200);

console.log(p1.price); // 1000 (Vẫn giữ nguyên, an toàn!)
console.log(p2.price); // 1200
```

### ⚖️ So Sánh Ưu/Nhược
| | Ưu điểm | Nhược điểm |
|---|---|---|
| **OOP** | **Trực quan**: Dễ hiểu vì ánh xạ 1-1 với thực tế (vật có tính chất, hành động). <br> **Kiểm soát**: Setter kiểm soát chặt chẽ việc thay đổi dữ liệu. | **Khó test**: Cần khởi tạo object với trạng thái cụ thể. <br> **Side-effects**: Thay đổi state ngầm có thể gây bug khó tìm. |
| **FP** | **Dễ test**: Chỉ cần input -> output, không cần mock object. <br> **Predictable**: Kết quả luôn giống nhau với cùng input. | **Dài dòng**: Cần định nghĩa nhiều type và hàm rời rạc. <br> **Bộ nhớ**: Tạo ra nhiều object mới thay vì sửa cái cũ (Performance overhead nếu không tối ưu). |

## 3. Bài Toán 2: Giỏ Hàng (State Management)

**Yêu cầu:** Thêm sản phẩm vào giỏ hàng.

### 🏛️ OOP: Stateful Service
Giỏ hàng là một "Hộp đen" lưu trữ trạng thái.

```typescript
class Cart {
  private items: Product[] = [];

  addItem(product: Product) {
    this.items.push(product); // ⚠️ Side-effect: Thay đổi mảng nội tại
  }

  getTotal() {
    return this.items.reduce((sum, p) => sum + p.price, 0);
  }
}

const myCart = new Cart();
myCart.addItem(p1);
// Khó để biết trạng thái myCart ở các thời điểm khác nhau nếu không debug kỹ
```

### 🧪 FP: Immutable Data Flow
Giỏ hàng chỉ là một danh sách. Thao tác thêm vào tạo ra một danh sách mới. Rất giống Redux.

```typescript
type Cart = Product[];

const addToCart = (cart: Cart, product: Product): Cart => {
  return [...cart, product]; // Trả về mảng mới
};

const calculateTotal = (cart: Cart): number => {
  return cart.reduce((sum, p) => sum + p.price, 0);
};

const emptyCart: Cart = [];
const cartWithItem = addToCart(emptyCart, p1);
// emptyCart vẫn rỗng. Lịch sử thay đổi được bảo toàn.
```

### ⚖️ So Sánh Ưu/Nhược
| | Ưu điểm | Nhược điểm |
|---|---|---|
| **OOP** | **Gọn gàng**: Gọi `cart.addItem()` ngắn gọn, tự nhiên. <br> **Memory**: Hiệu quả hơn vì sửa trực tiếp trên vùng nhớ cũ (Mutate). | **Race conditions**: Nếu nhiều luồng cùng gọi `addItem`, kết quả có thể sai lệch. <br> **Khó debug**: Cần theo dõi lịch sử biến đổi của biến `items` theo thời gian. |
| **FP** | **Time-travel Debugging**: Dễ dàng quay lại trạng thái cũ (Redux style). <br> **Concurrency**: An toàn tuyệt đối khi chạy đa luồng. | **GC Pressure**: Tạo rác (Garbage) nhiều do liên tục clone object/array. <br> **Học thuật**: Cú pháp Spread `[...]` hoặc `immutability helpers` có thể lạ lẫm. |

## 4. Bài Toán 3: Tính Giảm Giá (Strategy vs HOF)

**Yêu cầu:** Áp dụng các loại giảm giá khác nhau (Theo %, Theo số tiền, VIP).

### 🏛️ OOP: Strategy Pattern
Dùng Interface và Class để đa hình hóa thuật toán.

```typescript
interface DiscountStrategy {
  apply(price: number): number;
}

class PercentageDiscount implements DiscountStrategy {
  constructor(private percent: number) {}
  apply(price: number) { return price * (1 - this.percent); }
}

class FixedDiscount implements DiscountStrategy {
  constructor(private amount: number) {}
  apply(price: number) { return Math.max(0, price - this.amount); }
}

class Checkout {
  constructor(private strategy: DiscountStrategy) {}
  
  getFinalPrice(price: number) {
    return this.strategy.apply(price);
  }
}

const checkout = new Checkout(new PercentageDiscount(0.1)); // Giảm 10%
```

### 🧪 FP: Higher-Order Functions (HOF)
Hàm trả về hàm. Functions là "first-class citizens".

```typescript
// Định nghĩa kiểu hàm giảm giá
type DiscountFn = (price: number) => number;

// HOF: Hàm tạo ra hàm
const createPercentageDiscount = (percent: number): DiscountFn => 
  (price) => price * (1 - percent);

const createFixedDiscount = (amount: number): DiscountFn => 
  (price) => Math.max(0, price - amount);

// Usage
const tenPercentOff = createPercentageDiscount(0.1);
const minus50k = createFixedDiscount(50000);

console.log(tenPercentOff(1000)); // 900
```

### ⚖️ So Sánh Ưu/Nhược
| | Ưu điểm | Nhược điểm |
|---|---|---|
| **OOP** | **Cấu trúc rõ**: Dễ dàng tìm thấy logic trong các file class riêng biệt. <br> **Mở rộng**: Dễ thêm Strategy mới mà không sửa code cũ (Open/Closed Principle). | **Boilerplate**: Phải viết nhiều code (Interface, Class, Constructor) cho logic đơn giản. <br> **Phức tạp**: Cần hiểu Design Patterns. |
| **FP** | **Ngắn gọn**: Giảm thiểu code thừa, logic tập trung. <br> **Linh hoạt**: Có thể tạo ra logic mới ngay tại chỗ (Runtime) bằng cách kết hợp hàm. | **Type Safety**: Type của hàm `(number) => number` đôi khi quá chung chung, khó biết hàm đó làm gì nếu không đặt tên tốt. |

## 5. Bài Toán 4: Quy Trình Xử Lý Đơn Hàng (Chaining vs Composition)

**Yêu cầu:** Một đơn hàng cần qua các bước: `Validate` -> `Tính toán` -> `Lưu DB` -> `Gửi Email`.

### 🏛️ OOP: Chaining / Service Layer
Thường gọi các method lồng nhau hoặc liên tiếp trong một Service.

```typescript
class OrderService {
  process(order: Order) {
    if (!order.isValid()) throw new Error("Invalid"); // 1
    
    order.calculateTotal(); // 2
    
    this.repo.save(order); // 3
    this.mailer.send(order); // 4
  }
}
```

### 🧪 FP: Function Composition (Pipe)
Ghép nối các hàm nhỏ thành một quy trình lớn. Dữ liệu chảy qua các ống (pipes).

```typescript
// Giả sử ta có các hàm nhỏ thuần khiết (hoặc gần thuần khiết)
const validate = (o: Order): Order => { /*...*/ return o; };
const calculate = (o: Order): Order => { /*...*/ return o; };
const save = (o: Order): Order => { /*...*/ return o; };

// Helper để pipe (hoặc dùng thư viện như lodash/fp, ramda)
const pipe = (...fns: Function[]) => (x: any) => fns.reduce((v, f) => f(v), x);

// Tạo quy trình xử lý mới từ các khối nhỏ
const processOrder = pipe(
  validate,
  calculate,
  save
  // sendEmail (cần xử lý side-effect cẩn thận hơn trong FP thực tế, ví dụ dùng Monad)
);

processOrder(myOrder);
```

### ⚖️ So Sánh Ưu/Nhược
| | Ưu điểm | Nhược điểm |
|---|---|---|
| **OOP** | **Dễ đọc luồng**: Đọc code từ trên xuống dưới thấy rõ các bước. <br> **Tooling**: IDE hỗ trợ nhắc lệnh (IntelliJ) cực tốt cho Method Chaining. | **Coupling**: `OrderService` dễ bị phình to (God Class) và phụ thuộc chặt vào `this`. <br> **Khó tái sử dụng**: Method `validate` gắn liền với `OrderService`, khó bưng sang chỗ khác dùng riêng. |
| **FP** | **Tái sử dụng cao**: Các hàm `validate`, `save` đứng độc lập, dùng đâu cũng được. <br> **Test từng phần**: Test riêng từng hàm nhỏ dễ hơn test cả quy trình lớn. | **Khó debug**: Stack trace khi lỗi trong `pipe` thường rất khó đọc. <br> **Rối rắm**: Nếu lạm dụng `pipe/compose` quá nhiều tầng, code trở nên trừu tượng khó hiểu. |

## 6. Tổng Kết: Khi Nào Dùng Gì?

### ✅ Chọn OOP khi:
1.  **Domain Modeling phức tạp**: Các thực thể có mối quan hệ chặt chẽ, logic nghiệp vụ gắn liền với dữ liệu (DDD - Domain Driven Design thường hợp với OOP).
2.  **Quản lý UI Component có state phức tạp**: Dù React giờ dùng Hooks (FP), nhưng tư duy đóng gói Component vẫn mang hơi hướng OOP.
3.  **Thư viện/Framework ép buộc**: Ví dụ NestJS, Angular được thiết kế theo OOP (Dependency Injection class-based).

### ✅ Chọn FP khi:
1.  **Data Transformation**: Xử lý dữ liệu thuần túy (ETL, map/filter/reduce mảng lớn).
2.  **Concurrency / Parallelism**: Vì Immutability giúp tránh race conditions (không ai sửa dữ liệu của ai).
3.  **Logic tính toán phức tạp**: Cần test kỹ càng (Unit test pure function cực dễ).
4.  **React Hooks / Utilities**: Helper functions nên viết theo kiểu FP để dễ tái sử dụng.

### 🚀 Lời khuyên cho dự án E-commerce hiện đại (Next.js/TS):
**"Functional Core, Imperative Shell"**

-   Dùng **FP** cho logic nghiệp vụ, tính toán giá, validate, utility functions (trong thư mục `utils`, `lib`).
-   Dùng **OOP** (hoặc Class) cho các Service kết nối Database, API Client, tích hợp bên thứ 3 (để tận dụng Dependency Injection và quản lý cấu hình).
