# OOP Deep Dive: Từ Clean Code đến Design Patterns (E-commerce Context)

> Tài liệu chuyên sâu về Lập trình hướng đối tượng (OOP), các nguyên lý SOLID và Design Patterns ứng dụng trong thực tế dự án E-commerce.

---

## 1. 4 Trụ Cột Cơ Bản (The 4 Pillars)

Trước khi đi xa, hãy ôn lại nền tảng giúp OOP trở nên mạnh mẽ.

### 🛡️ Encapsulation (Tính Đóng Gói)
Bảo vệ dữ liệu, chỉ cho phép truy cập qua các "cổng" (method) được kiểm soát.

```typescript
class Wallet {
  private _balance: number = 0;

  deposit(amount: number) {
    if (amount <= 0) throw new Error("Tiền nạp phải dương");
    this._balance += amount;
  }

  get balance() { return this._balance; }
}
```

### 🧬 Inheritance (Tính Kế Thừa)
Tái sử dụng code từ lớp cha.

```typescript
class Employee {
  constructor(public name: string) {}
  work() { console.log("Working..."); }
}

class Manager extends Employee {
  manage() { console.log("Managing..."); }
}
```

### 🎭 Abstraction (Tính Trừu Tượng)
Ẩn chi tiết phức tạp, chỉ lộ ra những gì cần thiết.

```typescript
abstract class PaymentGateway {
  abstract charge(amount: number): void;
}
// User chỉ cần biết gọi hàm charge(), không cần biết bên dưới gọi API nào.
```

### 🦎 Polymorphism (Tính Đa Hình)
Một hành động có thể được thực hiện theo nhiều cách khác nhau tùy đối tượng.

## 2. SOLID Principles: Kim Chỉ Nam Cho Clean Code

5 nguyên lý giúp code dễ bảo trì, dễ mở rộng và tránh "Code thối".

### S - Single Responsibility Principle (SRP)
> **"Một class chỉ nên có một lý do duy nhất để thay đổi."**

❌ **Vi phạm:**
```typescript
class Order {
  calculateTotal() { /*...*/ }
  saveToDB() { /*...*/ } // Dính logic DB
  sendEmail() { /*...*/ } // Dính logic Email
}
```

✅ **Tuân thủ:** Tách nhỏ trách nhiệm.
```typescript
class Order { /* Chứa data & business logic */ }
class OrderRepository { /* Lưu DB */ }
class NotificationService { /* Gửi Email */ }
```

### O - Open/Closed Principle (OCP)
> **"Mở để mở rộng, đóng để sửa đổi."**

Bạn muốn thêm tính năng mới? Hãy viết class mới kế thừa/implement class cũ, ĐỪNG sửa code đang chạy ổn định.

❌ **Vi phạm:** Sửa hàm `calculate` mỗi khi có loại giảm giá mới.
```typescript
function calculate(type) {
  if (type === 'NEW') { /*...*/ }
  else if (type === 'VIP') { /*...*/ } // Phải sửa code cũ
}
```

✅ **Tuân thủ:** Dùng Polymorphism (Xem Strategy Pattern bên dưới).

### L - Liskov Substitution Principle (LSP)
> **"Class con phải thay thế được Class cha mà không làm hỏng chương trình."**

Đừng ép con phải làm những việc mà cha nó làm được nhưng nó thì không (hoặc ngược lại).

❌ **Vi phạm:**
```typescript
class Bird { fly() {} }
class Penguin extends Bird {
  fly() { throw new Error("Không bay được!"); } // Phá vỡ liskov
}
```

### I - Interface Segregation Principle (ISP)
> **"Thà dùng nhiều interface nhỏ còn hơn 1 interface lớn."**

Đừng bắt client implement những method họ không dùng.

❌ **Vi phạm:**
```typescript
interface User {
  login(): void;
  adminDeleteUser(): void; // User thường không cần cái này
}
```

✅ **Tuân thủ:** Tách `AdminInterface` riêng.

### D - Dependency Inversion Principle (DIP)
> **"1. Các module cấp cao không nên phụ thuộc vào các module cấp thấp. Cả hai nên phụ thuộc vào sự trừu tượng (Abstraction/Interface). 2. Sự trừu tượng không nên phụ thuộc vào chi tiết, mà chi tiết nên phụ thuộc vào sự trừu tượng."**

Hiểu đơn giản: Đừng để "Sếp" (High-level) phải đi hỏi tận tay "Nhân viên" (Low-level) làm thế nào. Sếp chỉ cần đưa ra yêu cầu (Interface), còn nhân viên cụ thể nào làm cũng được.

❌ **Cách làm tệ (Tightly Coupled - Phụ thuộc chặt):**
`OrderService` trực tiếp khởi tạo `MySQLDatabase`. Nếu sau này muốn đổi sang `PostgreSQL`, bạn phải sửa code ở khắp mọi nơi.
```typescript
class MySQLDatabase {
  save(order: any) { /* Lưu vào MySQL */ }
}

class OrderService {
  private db = new MySQLDatabase(); // ❌ Phụ thuộc trực tiếp vào Implementation

  process(order: any) {
    this.db.save(order);
  }
}
```

✅ **Cách làm chuẩn (DIP - Loose Coupling):**
`OrderService` chỉ phụ thuộc vào interface `IDatabase`. Nó không quan tâm bên dưới là MySQL hay MongoDB.

```typescript
// 1. Định nghĩa sự trừu tượng (Interface)
interface IDatabase {
  save(order: any): void;
}

// 2. Các implementation cụ thể (Chi tiết)
class MySQLDatabase implements IDatabase {
  save(order: any) { console.log("Lưu vào MySQL"); }
}

class MongoDatabase implements IDatabase {
  save(order: any) { console.log("Lưu vào MongoDB"); }
}

// 3. Module cấp cao (OrderService)
class OrderService {
  // ✅ Nhận vào Interface qua Constructor (Dependency Injection)
  constructor(private db: IDatabase) {}

  process(order: any) {
    this.db.save(order);
  }
}

// Sử dụng: "Bơm" (Inject) database nào tùy thích
const service = new OrderService(new MongoDatabase()); 
```

#### 🚀 Lợi ích của DIP:
1.  **Dễ dàng thay đổi**: Đổi database, đổi thư viện gửi mail, đổi cổng thanh toán chỉ bằng cách tạo class mới và inject vào, không cần sửa logic nghiệp vụ chính.
2.  **Dễ Mock để Test**: Khi viết Unit Test, bạn có thể tạo một `MockDatabase` đơn giản để truyền vào `OrderService` mà không cần bật database thật lên.
3.  **Song song hóa**: Team A viết `OrderService` dựa trên Interface, Team B viết `MySQLDatabase` dựa trên cùng Interface đó. Hai team làm việc độc lập mà không cần chờ nhau.

## 3. Design Patterns Trong E-commerce (Thực Chiến)

Design Patterns là các giải pháp mẫu cho các vấn đề phổ biến.

### 🏭 1. Factory Pattern (Creational)
**Bài toán:** Tạo các đối tượng Payment (Paypal, Stripe, Momo) dựa trên lựa chọn của user.

```typescript
interface PaymentProcessor {
  pay(amount: number): void;
}

class StripePayment implements PaymentProcessor {
  pay(amount: number) { console.log(`Thanh toán ${amount} qua Stripe`); }
}

class MomoPayment implements PaymentProcessor {
  pay(amount: number) { console.log(`Thanh toán ${amount} qua Momo`); }
}

class PaymentFactory {
  static create(type: 'stripe' | 'momo'): PaymentProcessor {
    if (type === 'stripe') return new StripePayment();
    if (type === 'momo') return new MomoPayment();
    throw new Error("Loại thanh toán không hỗ trợ");
  }
}

// Sử dụng
const payment = PaymentFactory.create('momo');
payment.pay(100);

#### 💡 Cái hay của Factory Pattern:
1.  **Che giấu sự phức tạp**: Client không cần biết logic khởi tạo `new` phức tạp bên trong, chỉ cần yêu cầu "thứ mình muốn".
2.  **Dễ mở rộng**: Khi có phương thức thanh toán mới (VD: VNPay), bạn chỉ cần sửa ở `Factory`, code ở các Controller/Service sử dụng không hề thay đổi.
3.  **Loose Coupling**: Tách biệt logic **Sử dụng** đối tượng và logic **Tạo** đối tượng.
```

### 👑 2. Singleton Pattern (Creational)
**Bài toán:** Chỉ muốn có **DUY NHẤT 1 instance** kết nối Database hoặc Config trong toàn bộ ứng dụng.

```typescript
class Database {
  private static instance: Database;
  
  private constructor() { console.log("Kết nối DB..."); }

  static getInstance(): Database {
    if (!Database.instance) {
      Database.instance = new Database();
    }
    return Database.instance;
  }
}

const db1 = Database.getInstance();
const db2 = Database.getInstance();
print(db1 === db2); // true (Cùng 1 object)

#### 💡 Cái hay của Singleton Pattern:
1.  **Tiết kiệm tài nguyên**: Tránh việc mở hàng chục kết nối Database hoặc đọc file Config lặp đi lặp lại.
2.  **Tính nhất quán**: Đảm bảo mọi nơi trong hệ thống đều đang dùng chung một "nguồn sự thật" (Single Source of Truth).
3.  **Kiểm soát truy cập**: Dễ dàng quản lý việc ghi log hoặc caching tập trung.
```

### 🎯 3. Strategy Pattern (Behavioral)
**Bài toán:** Thay đổi thuật toán tính phí vận chuyển (Giao nhanh, Giao chuẩn, Grab) ngay lúc chạy (runtime) mà không sửa code. **(Tuân thủ OCP)**.

```typescript
interface ShippingStrategy {
  calculate(weight: number): number;
}

class StandardShipping implements ShippingStrategy {
  calculate(weight: number) { return weight * 10; }
}

class ExpressShipping implements ShippingStrategy {
  calculate(weight: number) { return weight * 20 + 50; } // Đắt hơn
}

class ShippingContext {
  constructor(private strategy: ShippingStrategy) {}
  
  setStrategy(strategy: ShippingStrategy) {
    this.strategy = strategy;
  }
  
  execute(weight: number) {
    return this.strategy.calculate(weight);
  }
}

// Sử dụng
const shipping = new ShippingContext(new StandardShipping());
console.log(shipping.execute(10)); // 100

shipping.setStrategy(new ExpressShipping()); // Đổi chiến lược sang hỏa tốc
print(shipping.execute(10)); // 250

#### 💡 Cái hay của Strategy Pattern:
1.  **Loại bỏ If-Else chùm**: Thay vì viết hàng chục câu `if (type === 'fast') ...`, bạn tách mỗi logic vào một class riêng sạch sẽ.
2.  **Tuân thủ OCP**: Bạn có thể thêm 100 loại phí vận chuyển mới mà không cần động vào 1 dòng code cũ của `ShippingContext`.
3.  **Thay đổi linh hoạt**: Có thể đổi thuật toán ngay khi chương trình đang chạy (Runtime) dựa trên lựa chọn của User.
```

### 📡 4. Observer Pattern (Behavioral)
**Bài toán:** Khi một sản phẩm có hàng lại (Back in stock), tự động thông báo cho tất cả khách hàng đang chờ.

```typescript
interface Observer {
  update(productId: string): void;
}

class Customer implements Observer {
  constructor(private name: string) {}
  update(productId: string) {
    console.log(`🙋 ${this.name}: Yay! Sản phẩm ${productId} đã có hàng!`);
  }
}

class ProductSubject {
  private observers: Observer[] = [];

  subscribe(observer: Observer) {
    this.observers.push(observer);
  }

  notify(productId: string) {
    console.log(`📢 Thông báo: Sản phẩm ${productId} đã về kho!`);
    this.observers.forEach(obs => obs.update(productId));
  }
}

// Sử dụng
const iphone = new ProductSubject();
const user1 = new Customer("Alice");
const user2 = new Customer("Bob");

iphone.subscribe(user1);
iphone.subscribe(user2);

iphone.notify("IP15-PRO-MAX"); 
// Cả Alice và Bob đều nhận được thông báo

#### 💡 Cái hay của Observer Pattern:
1.  **Phản ứng tức thì**: Giúp hệ thống có tính tương tác cao (Reactive), không cần Client phải đi "hỏi" liên tục (Polling).
2.  **Độc lập đối tượng**: `ProductSubject` không cần biết `Customer` là ai, nó chỉ cần biết danh sách cần gửi thông báo. Bạn có thể thêm `SmsService`, `EmailService` làm Observer rất dễ dàng.
3.  **Phát tán tin tin**: Một sự kiện có thể kích hoạt chuỗi hành động ở nhiều module khác nhau một cách tự động.
```

### 🔌 5. Adapter Pattern (Structural)
**Bài toán:** Hệ thống cũ dùng `OldLogger` (xml), hệ thống mới cần `NewLogger` (json). Cần một "bộ chuyển đổi" để code mới chạy được với thư viện cũ.

```typescript
// Interface mới mong muốn
interface ILogger {
  logJSON(msg: string): void;
}

// Thư viện cũ (3rd party) không sửa được
class OldLogger {
  logXML(msg: string) { console.log(`<log>${msg}</log>`); }
}

// Adapter: Biến cái Cũ thành cái Mới
class LoggerAdapter implements ILogger {
  constructor(private oldLogger: OldLogger) {}

  logJSON(msg: string) {
    // Chuyển đổi logic nếu cần
    this.oldLogger.logXML(msg); 
  }
}

const adapter = new LoggerAdapter(new OldLogger());
adapter.logJSON("System Error"); // Output: <log>System Error</log>

#### 💡 Cái hay của Adapter Pattern:
1.  **Tái sử dụng code cũ**: Không cần đập đi xây lại những thư viện cũ/thư viện bên thứ 3 vẫn đang chạy tốt.
2.  **Khả năng tương thích**: Giúp các module có Interface khác nhau vẫn có thể "nói chuyện" được với nhau.
3.  **Nguyên tắc Single Responsibility**: Tách biệt code logic nghiệp vụ khỏi code chuyển đổi định dạng dữ liệu phức tạp.
```

## 4. Tổng Kết

- **Encapsulation & Abstraction**: Giúp che giấu sự phức tạp.
- **Inheritance & Polymorphism**: Giúp tái sử dụng và mở rộng linh hoạt.
- **SOLID**: Giữ cho kiến trúc phần mềm sạch sẽ, bền vững.
- **Design Patterns**: Là những công thức đã được kiểm chứng để giải quyết vấn đề cụ thể - đừng lạm dụng, hãy dùng đúng chỗ.
